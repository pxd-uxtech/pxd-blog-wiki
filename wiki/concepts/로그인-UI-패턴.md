# 로그인 UI 패턴

**로그인 UI**는 서비스 진입의 첫 관문으로, 인증 방식 선택이 사용자 경험에 직접 영향을 미친다. 어떤 정보를 기반으로 로그인하게 할 것인지가 핵심 설계 결정이다.

**ID + Password 방식**은 국내 웹사이트에서 가장 일반적이다. 사용자가 ID를 별도로 기억해야 하는 부담이 있고, 사이트마다 ID 규칙(자릿수, 시작 문자 제한 등)이 달라 한 사람이 여러 ID를 운용하게 된다. 이메일을 별도 입력해야 하는 추가 부담도 있다.

**이메일 + Password 방식**은 해외 서비스에서 표준화된 방식이다. 이메일 하나만 기억하면 되는 장점이 있지만, 주소 입력이 ID보다 길고 여러 이메일을 사용하는 경우 어느 이메일로 가입했는지 혼동할 수 있다. **모바일에서는 이메일 입력이 특히 불편**하다는 점도 고려해야 한다. 이메일 중복 체크는 필수이고, 이메일 인증 여부는 "사용자 불편 감수 vs. 입력 오류 책임 전가" 사이의 가치 판단이 필요하다.

**이메일 또는 ID 선택 방식**은 두 가지를 모두 지원하되 복잡성이 높아 보조적 방식으로 적용하는 것이 적합하다.

**소셜 로그인(OAuth)** 방식은 Facebook, Twitter, Google 계정으로 별도 회원가입 없이 인증하는 방법이다. 개인정보가 많이 필요 없는 서비스에서 활발히 활용된다. Open ID 방식은 점차 줄어들고 소셜 OAuth가 표준화되는 추세다.

**전화번호 인증 방식**은 모바일 앱에 적합하다. 카카오톡이 대표 사례로, 회원가입 없이 인증이 가능하지만 전화번호 변경 시 계정 연속성이 끊기는 문제가 있다. 두 개의 전화번호를 사용하는 경우나 다른 서비스와의 연동에도 제약이 따른다.

## 핵심 내용
- **이메일 + Password**가 단일 기억 항목, 중복 방지 측면에서 가장 균형적
- ID 방식: 규칙 다양성으로 사용자가 여러 ID 관리 → 이메일 병행 필요
- 이메일 인증: 입력 편의 vs. 오류 방지의 가치 판단이 선행
- 소셜 로그인: 개인정보 최소 수집 서비스에서 효과적
- 전화번호 인증: 모바일 전용, 번호 변경 시 계정 연속성 단절 문제
- **Log in vs. Sign in** 용어 선택도 서비스 톤 설계의 일부
