# 팝업 UI 패턴

**팝업(Pop-up)**은 화면 흐름을 일시 중단시키고 사용자의 주의를 특정 정보나 선택으로 유도하는 UI 요소다. Moux 연구회 백승화의 저서 『관계디자인』은 팝업을 **확인팝업**, **알림팝업**, **입력팝업**, **옵션팝업** 네 가지 유형으로 분류하고, 각 유형을 다시 발생 원인에 따라 **인터럽트**(기기 내부 상황에 의한 발생)와 **이벤트**(사용자 동작에 대한 반응)로 구분했다. pxd 블로그는 이 분류를 바탕으로 유형별 팝업이 실제로 어떻게 설계되어야 하는지 구체적인 개선 기준을 정리했다.

**소프트웨어 인터럽트 팝업**(오류 팝업)은 흔히 "버그니까 사용자가 볼 일이 적다"고 가볍게 취급되지만, 실제로는 신중한 설계가 필요하다. "알 수 없는 오류가 발생했습니다" 류의 팝업은 무엇이 문제인지, 어떻게 해결해야 하는지 알려주지 않아 사용자가 오류 메시지를 그대로 복사해 웹에서 검색하게 만든다. 이런 팝업을 설계할 때는 "이 인터럽트 상황이 정말 사용자가 알아야 할 정보인가", "사용자가 이해할 수 있는 언어로 어떻게 표현할 것인가"를 먼저 물어야 한다.

**하드웨어 인터럽트에 대한 알림팝업**은 단순 확인만 요구하는 경우가 많은데, 이럴 때는 모달형 알림 대신 화면을 가리지 않고 짧게 나타났다 사라지는 **토스트 알림(Toast Notification)**이 더 적절하다. **이벤트 동작재확인 팝업**(삭제 확인 등)은 사용자가 정보를 잃을 실질적 위험이 있을 때만 노출해야 한다는 원칙이 핵심이다. 백업 수단(예: 휴지통)이 있는 삭제 동작에는 재확인 팝업을 생략해 조작을 가볍게 하고, 반대로 삭제 버튼이 곧바로 노출되어 실수로 누르기 쉬운 UI에는 재확인 팝업으로 안전장치를 둔다. 즉 팝업 노출 여부는 "되돌릴 수 있는가"라는 되돌림 가능성(reversibility) 기준으로 취사선택해야 한다.

이 사례들을 종합하면 팝업 설계의 공통 원칙은 세 가지로 정리된다. 첫째, 가능하면 팝업을 아예 띄우지 않는 것이 최선이며 꼭 필요할 때만 정확하게 사용한다. 둘째, 팝업에 쓰이는 용어는 개발자·시스템 언어가 아니라 사용자가 이해할 수 있는 언어로 표현해야 한다. 셋째, 팝업 전후의 시나리오(트리거 조건, 후속 동작)를 간결하게 설계해 불필요한 인터럽트를 줄여야 한다.

## 핵심 내용
- 팝업 4대 유형: **확인팝업·알림팝업·입력팝업·옵션팝업**, 발생 원인은 **인터럽트**(시스템)와 **이벤트**(사용자 동작)로 구분
- 소프트웨어 인터럽트(오류) 팝업: "무엇이 문제인지 + 어떻게 해결하는지"를 사용자 언어로 설명해야 함
- 하드웨어 인터럽트 알림은 단순 확인용이면 모달보다 **토스트 알림**이 적합
- 이벤트 재확인 팝업은 **되돌림 가능성**(백업/휴지통 유무)을 기준으로 노출 여부 결정
- 공통 원칙: 팝업은 최소화, 사용자 언어 사용, 전후 시나리오 간결화
